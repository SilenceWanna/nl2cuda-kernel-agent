## 环境基线（已确认）

- 开发机：Windows 11，无 NVIDIA GPU，仅写代码 + git。
- GPU 环境：Google Colab（免费版 Tesla T4，sm_75，nvcc 12.8，PyTorch 2.11.0+cu128，numpy 2.0.2）。GPU 配置非当前重点，后续如需可升级。
- 协作方式：代码放 GitHub **public** 仓库（https://github.com/SilenceWanna/nl2cuda-kernel-agent），Colab 里 clone/pull 运行（无需认证），输出回贴。
- 验收用例（已定）：**成对距离 / RBF 高斯核矩阵**（非 attention），FP32，形状 **N=M（默认 2048，env `RBF_SIZE` 可切 4096）, D=64**。
  - 结构：X:[N,D], Y:[M,D] → 距离 `Dist[i,j]=||x_i-y_j||²`，RBF 核 `K[i,j]=exp(-gamma·Dist[i,j])`；输出 K:[N,M]；反向求 dX、dY。
  - **形状说明**：T4(16GB) 下 4096² 广播反向 OOM（中间量 4GB，峰值~12GB），故默认缩到 2048²（中间量~1GB，峰值~3-4GB）；升级到 ≥24GB 卡后设 `RBF_SIZE=4096` 恢复，代码不变。赢面逻辑与形状无关。
  - **赢面关键前提**：PyTorch 参考实现必须写成**自然广播形式** `(x[:,None]-y[None]).square().sum(-1)`（torch.compile 生成通用 Triton，缺 GEMM 式 D 倍数据复用，可被手写 tiling kernel 打败）；**不能**写成 `||x||²+||y||²-2·XYᵀ` 分解（会走 cuBLAS GEMM，难赢）。用广播形式是正当的（自然语言最直接翻译）。
- T4 约束：不支持 TF32；FP32 路径优化靠 tiling / float4 / warp 原语 / 算子融合。

## 阶段 7 — 用户输入极简化（方法论内化进 agent 约定文件）

**目标**：用户在 agent 交互界面**只输入算法定义**（等价于现在 `description.md` 的内容：自然语言 + 前向数学 + shape/dtype + 对哪些输入求梯度），agent 就自动按 skill 走完"建 case → 写 reference/kernel/op → 自测 → 优化 → 达标"全流程——**不再手输"读 SKILL.md、自主推导反向、守防作弊红线、自测、稳定过线"那一大段 kickoff prompt**。把测试中反复手输的要求**内化进各 agent 自动加载的约定文件**。

**可行性（2026-07-13 评估）**：高度可行，主要是"文档工程"。关键——不同 agent 的"自动加载"机制不同：
- **Claude Code**：`CLAUDE.md` 自动注入上下文。
- **aider**：`CONVENTIONS.md`（可 auto-load）+ `.aider.conf.yml`（预设 model / `--auto-test` / `--test-cmd`）。
- **codex**：`AGENTS.md`（codex 自动读）。
- **gptme**：无标准约定文件，靠启动脚本预置 `--system` 自定义 system prompt（内化程度最弱）。
- 边界：完全零输入不现实（用户至少给算法定义，合理）；case 命名歧义时约定"不确定就问一句"。
