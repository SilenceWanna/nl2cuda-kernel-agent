## 环境基线（已确认）

- 开发机：Windows 11，无 NVIDIA GPU，仅写代码 + git。
- GPU 环境：Google Colab（免费版 Tesla T4，sm_75，nvcc 12.8，PyTorch 2.11.0+cu128，numpy 2.0.2）。GPU 配置非当前重点，后续如需可升级。
- 协作方式：代码放 GitHub public 仓库（https://github.com/SilenceWanna/nl2cuda-kernel-agent），Colab 里 clone/pull 运行，输出回贴。
- 验收用例（初定）：标准 causal Multi-Head Attention，FP32，形状固定（batch/heads/seq_len/head_dim 待阶段 0 敲定）。
- T4 约束：不支持 TF32；FP32 路径优化靠 tiling / float4 / warp 原语 / 算子融合。

## 阶段 0 — 项目地基与验收基础设施（GPU 无关，可先写；在 Colab 验证）

- [ ] **0.1** 建立项目骨架：目录结构（`src/` `tests/` `benchmarks/` `kernels/` `notebooks/`）、`README.md`、依赖清单 `requirements.txt`（torch/numpy 等）。
- [ ] **0.2** 敲定验收用例的精确规格：写 `config.py`，固定 causal MHA 的 batch、heads、seq_len、head_dim、dtype、随机种子列表（≥5 个）。
- [ ] **0.3** 手写 PyTorch 参考实现 `reference/mha_reference.py`：causal MHA 前向 + autograd 反向，作为正确性金标准（不依赖 SDPA/高层融合算子的"标准写法"，用基础 matmul+softmax+mask 表达）。
- [ ] **0.4** 正确性验证框架 `tests/test_correctness.py`：对给定 kernel 实现，用 ≥5 组随机种子跑前向 allclose(atol=rtol=1e-2) + 反向各梯度 allclose；小规模用 double `gradcheck`。设计成"传入一个实现对象即可复用"。
- [ ] **0.5** 计时基准框架 `benchmarks/bench.py`：warmup≥10、正式≥100 取几何均值、CUDA events 计时、每次前后 synchronize、3 次重跑算 CV，CV>5% 作废重测；前向/反向分别计时。**此脚本设计为只读、进程隔离、不被 agent 改动**。
- [ ] **0.6** Colab 驱动 notebook `notebooks/run.ipynb`：git clone/pull + 安装依赖 + 运行探测/验证/基准的入口。跑通"环境探测"一段（复用 `scripts/probe_env.py`），确认 notebook 回环可用。

## 阶段 1 — 手写 CUDA Kernel 打通全链路（验证验收管线本身可用）

- [ ] **1.1** CUDA 扩展加载脚手架 `kernels/loader.py`：用 `torch.utils.cpp_extension.load` 即时编译 `.cu`，编译参数针对 sm_75；封装成 python 可调用。
- [ ] **1.2** 手写 causal MHA **前向** kernel（朴素正确版，非优化）`kernels/mha_forward.cu`：先追求正确，不追求快。
- [ ] **1.3** 前向接入正确性框架：前向 allclose 对拍参考实现通过（≥5 组种子）。
- [ ] **1.4** 手写 causal MHA **反向** kernel（朴素正确版）`kernels/mha_backward.cu`：计算 dQ/dK/dV。
- [ ] **1.5** 反向接入正确性框架：反向各梯度 allclose + 小规模 gradcheck 通过。
- [ ] **1.6** 首次基准对比：手写 kernel vs `torch.compile`(默认 mode) 前反向分别计时，记录 baseline 数字（此时大概率还打不过，用于摸清差距）。

## 阶段 2 — Agent 化（把人工流程自动化）

- [ ] **2.1** Agent 架构设计文档 `docs/agent_design.md`：模块划分（NL 解析 → PyTorch 生成 → CUDA 生成 → 编译/验证/profile 反馈循环）、接口约定、失败重试策略。
- [ ] **2.2** NL → PyTorch 参考实现生成模块：输入自然语言 + shape/dtype，产出可运行 PyTorch 参考。用阶段 0 的手写参考作为回归对照。
- [ ] **2.3** PyTorch → CUDA kernel 生成模块（前向）：产出 `.cu`，自动接正确性框架校验，失败则带编译/数值误差反馈重试。
- [ ] **2.4** PyTorch → CUDA kernel 生成模块（反向）：同上，产出反向 kernel 并校验。
- [ ] **2.5** 端到端闭环串联：一条命令从 NL 描述跑到"正确性通过的前反向 kernel"。

## 阶段 3 — 优化冲刺（打赢 torch.compile ≥5%）

- [ ] **3.1** 引入 profile 反馈：用 Nsight/torch profiler 找瓶颈（访存/占用率/bank conflict）。
- [ ] **3.2** 前向优化迭代：shared-memory tiling + float4 向量化 + warp 原语 + 算子融合，逐步逼近并超过 torch.compile 前向。
- [ ] **3.3** 反向优化迭代：同类手段优化反向，超过 torch.compile 反向。
- [ ] **3.4** 达标确认：规范计时下前向、反向均 ≥1.05×，3 次重跑 CV≤5%，正确性仍通过。（如 T4 难以达标，评估升级 GPU 配置。）

## 阶段 4 — 交付

- [ ] **4.3** 最终 README 与使用说明，收尾。
