## 环境基线（已确认）

- 开发机：Windows 11，无 NVIDIA GPU，仅写代码 + git。
- GPU 环境：Google Colab（免费版 Tesla T4，sm_75，nvcc 12.8，PyTorch 2.11.0+cu128，numpy 2.0.2）。GPU 配置非当前重点，后续如需可升级。
- 协作方式：代码放 GitHub **public** 仓库（https://github.com/SilenceWanna/nl2cuda-kernel-agent），Colab 里 clone/pull 运行（无需认证），输出回贴。
- 验收用例（已定）：**成对距离 / RBF 高斯核矩阵**（非 attention），FP32，形状 **N=M（默认 2048，env `RBF_SIZE` 可切 4096）, D=64**。
  - 结构：X:[N,D], Y:[M,D] → 距离 `Dist[i,j]=||x_i-y_j||²`，RBF 核 `K[i,j]=exp(-gamma·Dist[i,j])`；输出 K:[N,M]；反向求 dX、dY。
  - **形状说明**：T4(16GB) 下 4096² 广播反向 OOM（中间量 4GB，峰值~12GB），故默认缩到 2048²（中间量~1GB，峰值~3-4GB）；升级到 ≥24GB 卡后设 `RBF_SIZE=4096` 恢复，代码不变。赢面逻辑与形状无关。
  - **赢面关键前提**：PyTorch 参考实现必须写成**自然广播形式** `(x[:,None]-y[None]).square().sum(-1)`（torch.compile 生成通用 Triton，缺 GEMM 式 D 倍数据复用，可被手写 tiling kernel 打败）；**不能**写成 `||x||²+||y||²-2·XYᵀ` 分解（会走 cuBLAS GEMM，难赢）。用广播形式是正当的（自然语言最直接翻译）。
- T4 约束：不支持 TF32；FP32 路径优化靠 tiling / float4 / warp 原语 / 算子融合。

> 目的：证明"编译→正确→计时→对比 torch.compile"这条评测管线本身可用，并摸清 T4 上打赢 torch.compile 的难度；产物同时作为 skill 提示里的正确解示例。

- [x] **1.2** 手写 RBF 核 **前向** kernel（朴素正确版）`kernels/rbf_forward.cu`。(完成于 2026-07-07；一线程一输出 K[i,j]，含 `rbf_op.py` 封装，notebook 加 2.6 段)
- [x] **1.3** 前向接入验证器：allclose 对拍参考实现通过（≥5 组种子）。(完成于 2026-07-07：Colab 5 种子全 PASS，max_abs_err~2.4e-7)
- [x] **1.4** 手写 RBF 核 **反向** kernel（朴素正确版）`kernels/rbf_backward.cu`：计算 dX、dY。(完成于 2026-07-07；两 kernel 分别算 dX(沿j归约)/dY(沿i归约)，autograd.Function 包装，待 Colab 验证)

- [x] **1.6** 首次基准：手写 kernel vs torch.compile 前反向分别计时，记录 baseline（用于摸差距、指导阶段 3 优化方向）。(完成于 2026-07-07)

## 阶段 2 — 编写 Skill（方法论指令 + 工具，宿主无关 + **算法无关**）

> **通用性定位**：最终目标是 skill 对**任意算法结构**都能完成闭环；RBF 只是第一个验收用例，后续会加更多算法。因此架构分两层：**通用框架**（方法论、评测工具、编译加载、loop——算法无关）+ **算法 case**（NL 描述、参考实现、shape/dtype、金标准——每个算法一份，可替换）。RBF 是 `cases/` 下的第一个实例，不写死进框架核心。

- [x] **2.1** Skill 设计文档 `skill/DESIGN.md`：skill 的输入/输出、**通用框架 vs 算法 case 的分层与接口约定**、agent 每步该做什么、可用工具（验证器/基准器/加载器）、失败反馈如何回喂、防作弊红线。(完成于 2026-07-07：定义 Case 协议、两层架构、工具契约、8 节)
- [x] **2.2** 编写 `skill/SKILL.md`（方法论主体，**算法无关**）：指导宿主 agent 从任意 NL 描述 → PyTorch 参考 → CUDA 前反向 → 用验证器自检 → 用基准器测速的完整步骤与判据（自然语言，宿主无关）。(完成于 2026-07-07：Claude Code skill 格式 frontmatter + 7 步流程 + 防作弊红线 + 新增 case 指引)
- [x] **2.3** 通用工具/case 结构：把验证器、基准器、编译加载重构为**接受"算法 case"参数**的命令行工具（agent 一条命令可调，只读评测脚本），带清晰 stdout 反馈。把 RBF 归整为 `cases/rbf/`（参考实现 + config + 描述），作为第一个 case 模板。(完成于 2026-07-07；**Colab 回归通过**：前反向 5 种子全 PASS，数值与重构前完全一致，新 CLI 正常)
- [x] **2.4** loop 驱动脚本 `skill/loop.md`（或 py）：定义"生成→编译→验证→读 profile→改进"的迭代循环与终止条件（正确且前反向均达标 / 到达轮次上限），算法无关。(完成于 2026-07-07：循环体 + 3 终止条件 + 6 优化手段 + 纪律)
- [ ] **2.5** Skill 干跑测试：喂 RBF 核的 NL 描述，宿主 agent 跑到"正确性通过的前反向 kernel"（先不要求达速度标），验证通用框架在第一个 case 上闭环。

## 阶段 3 — 优化冲刺（由 skill/loop 驱动宿主 agent，打赢 torch.compile ≥5%）

- [ ] **3.1** profile 反馈接入 skill：让 agent 能读 profiler 输出（访存/占用率/bank conflict）并据此改 kernel。
- [ ] **3.2** 在 skill 指导下迭代**前向**优化：tiling + float4 + warp 原语 + 融合，逼近并超过 torch.compile 前向。
- [ ] **3.3** 在 skill 指导下迭代**反向**优化：同类手段超过 torch.compile 反向。
- [ ] **3.4** 达标确认：规范计时下前反向均 ≥1.05×，3 次重跑 CV≤5%，正确性仍通过。（如 T4 难达标，评估升级 GPU。）
